Повторная сверка отчёта с реальным кодом на диске. Проверено пофайлово, с номерами строк. Дата сверки: 13 июля 2026.

Вывод в двух словах: **аудит в целом добросовестный, но частично устарел.** Из 42 пунктов ~29 подтверждаются, 9 уже исправлены или неверны, а несколько с неточными номерами строк. И главное — аудит **пропустил один build-ломающий баг**, который сейчас важнее всего остального.

## 🚨 Пропущено аудитом — критично

### NEW-1: `digiseller/link/route.ts` синтаксически сломан
Файл `connect/app/api/shop/digiseller/link/route.ts`:

```
12   try {
...
92     return NextResponse.json({ success: true }, { headers })   // ← выход из функции
93     console.error('[Digiseller link] Error:', err);            // ← недостижимо + err не определён
94     return NextResponse.json({ success:false, error:'…' }, {status:500, headers})
95   }                                                            // ← закрывает try
96 }                                                              // ← закрывает функцию
```

`try` на строке 12 **не имеет `catch`/`finally`** — это `SyntaxError`. Плюс `err` на строке 93 нигде не объявлена, а строки 93–94 недостижимы (стоят после `return`).

Последствия: `next build` для всего проекта `connect` падает, привязка заказов Digiseller не работает. **Чинить в первую очередь.** Рекомендую сразу прогнать `npm run build` в `connect/` — это подтвердит.

### NEW-2: реальные утечки `err.message` — но в других файлах
Аудит (HIGH-6) указал на `ggsel/link:166`, но там как раз возвращается generic-сообщение. Настоящие утечки `err.message` клиенту здесь:
- `ggsel/send-message/route.ts:93`
- `ggsel/order-details/route.ts:126`

### NEW-3: `connect/vercel.json` — небезопасная связка CORS
Строки 6–7: `Access-Control-Allow-Credentials: true` вместе с `Access-Control-Allow-Origin: *`. Это хуже, чем просто `*`: сигнализирует о намерении пускать кросс-доменные запросы с куками. В сочетании с HIGH-1 (нет авторизации) — усиливает риск.
