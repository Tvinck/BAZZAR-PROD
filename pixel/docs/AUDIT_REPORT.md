# 🔴 КРИТИЧЕСКИЕ ОШИБКИ (исправить срочно)

1.  **Отсутствие проверки прав администратора**
    *   **Файл:** `server/routes/admin.js` (все эндпоинты)
    *   **Описание:** Эндпоинты админ-панели (метрики, рассылки, управление моделями) используют только мидлвар `authTG`, который проверяет валидность данных Telegram, но не проверяет, является ли пользователь администратором (поле `role` в БД).
    *   **Риск:** Любой пользователь Telegram Mini App может получить доступ к админ-панели и выполнять критические действия через API.

2.  **Небезопасная отладочная авторизация в "продакшене"**
    *   **Файл:** `server/middleware/auth.js`, строка 31
    *   **Описание:** Код проверяет `process.env.ALLOW_DEV_AUTH === 'true'`. Если эта переменная случайно будет включена в продакшене, любой сможет войти под любым `user_id` через заголовок `x-dev-auth-id`.
    *   **Риск:** Полный захват любого аккаунта.

3.  **Использование Service Role Key для клиентских запросов**
    *   **Файл:** `server/routes/user.js`, линия 10 (`/stats`)
    *   **Описание:** Эндпоинт открыто обходит RLS, используя привилегированный ключ Supabase через `botAnalytics`.
    *   **Риск:** Потенциальная утечка данных о других пользователях, если логика фильтрации в JS даст сбой.

# 🟡 ВАЖНЫЕ НЕДОЧЁТЫ (исправить в ближайшее время)

1.  **Отсутствие валидации входных данных**
    *   **Файлы:** `server/routes/auth.js`, `server/routes/user.js`, `server/routes/admin.js`
    *   **Описание:** Почти во всех эндпоинтах данные из `req.body` и `req.query` используются без предварительной валидации схемы (отсутствуют Zod, Joi или express-validator).
    *   **Риск:** Потенциальные ошибки сервера при некорректных данных, возможность инъекций в логику.

2.  **Крайне низкое покрытие тестами**
    *   **Папка:** `/tests` и `/server/__tests__`
    *   **Описание:** Покрыто менее 5% функционала. Всего 2 юнит-теста на бэкенде и 2 E2E сценария. Критические функции (платежи, генерация, админ-панель) не покрыты автотестами.

3.  **Использование бета-версии Express**
    *   **Файл:** `package.json`, строка 41 (`"express": "^5.2.1"`)
    *   **Описание:** Используется 5-я версия Express, которая всё еще находится в стадии активной разработки/миграции. Для стабильных проектов рекомендуется 4.x.

4.  **Хардкод URL в коде**
    *   **Файл:** `server/index.js`, строки 29-33
    *   **Описание:** CORS конфигурируется с захардкоженными URL Vercel. Это затрудняет перенос проекта на другие домены.

    *   Некоторые вызовы Supabase (например, получение UUID по `telegram_id`) повторяются во многих файлах вместо использования единого сервисного слоя.

3.  **Логирование**
    *   В проекте много `console.log`, но нет структурированного логирования (Winston/Pino) для удобного мониторинга в продакшене.

# ✅ ЧТО РАБОТАЕТ ХОРОШО

1.  **Архитектура API Gateway**
    *   Грамотное разделение на микросервисы (User, Payments, Generation) с использованием `http-proxy-middleware`. Это позволит легко масштабироваться.
2.  **Telegram-интеграция**
    *   Качественная реализация проверки подписи Telegram (`verifyTelegramWebAppData`).
3.  **Frontend UX**
    *   Использование современных библиотек для анимаций (`framer-motion`, `gsap`) и эффективное управление запросами через `react-query`.
4.  **Database Design**
    *   Схема данных нормализована, используются индексы на часто запрашиваемых полях (`telegram_id`), настроены базовые RLS политики.
